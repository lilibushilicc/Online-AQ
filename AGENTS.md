# 项目级规则

- 始终使用简体中文回复。
- 本规则仅针对当前项目 `D:\study\Systemlocation\Online-AQ` 生效。
- `docs/` — 项目说明文档、设计说明、功能说明、接口说明、使用文档。
- `docs/change-summary/` — 每次代码改动后的总结文档。
- `log/` — 运行日志、构建日志和排查记录。
- `release/package/` — 打包产物。
- `release/docker/` — Docker 相关文件与镜像构建产物。
- `WillPlan/` — 将来实施的计划、预研文件和后续规划资料。
- `Oline-AQ-vue/` 和 `Oline-AQ-spring/` 只保留项目源码及代码运行所需文件，不在这两个目录内新增长期沉淀文档、日志文档、规划文档和打包产物。
- 如需新增说明文档 → `docs/`；改动总结 → `docs/change-summary/`；日志 → `log/`；规划材料 → `WillPlan/`。

## 每次代码变更后的工作

每次对项目代码进行新增、修改或删除后，依次执行以下操作：

1. **检查现有文档** — 检查 `docs/` 中现有说明文档和 `README.md` 是否仍适用；内容过时、缺失或与当前实现不一致的，必须同步更新。**每个模块/功能应在 `docs/` 下有对应的说明文档**（如 `docs/用户模块.md`、`docs/考试模块.md`），修改对应功能时同步更新该文档。
2. **追加 README 更新记录** — 在 `README.md` 顶部「更新记录」节追加一条条目，格式为 `> YYYY-MM-DD 更新：改动简述`，多个条目按日期倒序排列。保持「更新记录」位于文件最顶部（标题下方第一块内容）。
3. **生成改动总结文档** — 按下方「改动总结文档结构」创建 Markdown 文档，统一存放在 `docs/change-summary/`。

### 改动总结文档命名

```
功能名称-改动总结.md
```

示例：`登录页面-改动总结.md`、`考试提交评分-改动总结.md`

如需按时间排序，可加 `YYYY-MM-DD-HHMM-` 前缀：`2026-05-17-1015-登录页面-改动总结.md`

### 改动总结文档应包含的视角

- 改动目标：本次为什么改。
- 用户视角：用户能看到什么变化，操作流程有什么变化。
- 前端视角：页面、组件、路由、状态管理、请求封装如何实现。
- 后端视角：Controller、Service、Mapper、实体、配置如何实现。
- 数据库视角：涉及哪些表、字段、数据流和存储变化。
- 接口视角：涉及哪些接口，请求参数、返回数据和调用关系是什么。
- 文件视角：新增、修改、删除了哪些关键文件，各自负责什么。
- 运行视角：如何启动、如何验证、执行过哪些构建或测试命令。
- 风险与后续：当前限制、潜在问题、后续可以继续优化什么。

## 优化与建设规划文档要求

- 每次提出优化方案或建设计划时，必须生成一份 Markdown 规划文档，统一存放在 `WillPlan/` 目录下。
- 规划文档命名格式：
  - 纯中文名称（推荐），如 `并发优化方案.md`
  - 如需按时间排序可加前缀 `YYYY-MM-DD-HHMM-`，如 `2026-05-23-1106-并发优化方案.md`
- 规划文档应包含目标、现状分析、分阶段方案、风险与后续等内容。
- **计划实施完成后，不删除已实施条目，而是在对应条目标题或行首添加删除线（`~~内容~~`）标记已实施**，保留完整的历史记录。
