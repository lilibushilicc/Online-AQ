# 登录失败修复总结

## 1. 改动目标

本次改动用于修复“登录失败”的实际根因之一：数据库中缺少默认测试账号时，前端会始终登录失败，即使用户名、密码和身份输入正确也无法通过认证。此前项目依赖手工执行 `Oline-AQ-spring/sql/init.sql` 导入用户数据，一旦数据库是空库、历史库数据丢失或只建表未导数，登录功能就不可用。

## 2. 用户视角

用户现在在完成数据库建表并启动后端后，即使数据库里没有默认测试用户，也能直接使用以下账号登录：

- 教师：`admin / 123456`
- 学生：`2023001 / 123456`

如果这两个账号已经存在，系统不会重复插入，也不会覆盖原有数据。

## 5. 数据库与数据流

本次未新增表和字段。

新的账号初始化数据流如下：

1. 应用启动。
2. `DemoUserInitializer` 查询 `sys_user` 表中是否存在 `admin` 和 `2023001`。
3. 若账号不存在，则自动插入默认账号。
4. 登录接口继续按原流程从 `sys_user` 表校验 `username`、`password`、`role`。

## 8. 验证方式

```sh
cd Oline-AQ-spring

```

验证结果：

- 后端编译通过。
- 由于当前环境未直接连接数据库执行联调，请在本地启动服务后再用默认账号验证一次真实登录。

## 9. 风险与后续优化

- 当前仅自动补齐默认账号，不会自动建表；首次部署仍需先执行 `Oline-AQ-spring/sql/init.sql`。
- 默认账号密码仍为明文示例值，仅适合开发和演示环境，后续建议接入密码加密存储。
- 后续可继续补充启动日志输出，明确提示“默认账号已创建”或“默认账号已存在”，便于排查。
