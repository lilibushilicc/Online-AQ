# 成绩列表按试卷聚合优化改动总结

## 1. 改动目标

本次改动同时针对 Android App (`Oline-AQ-app`) 和网页端 (`Oline-AQ-vue`) 的学生成绩页，目标是把原来按每次提交平铺的成绩列表改成按试卷聚合展示。

改动前，学生同一个试卷考了多次，成绩页会显示多条同名记录，找试卷和看历史都不方便。改动后，成绩页只显示一张试卷卡片，点进去再看到历次作答记录，再点某一次记录进入详细做题情况。

## 2. 用户视角

- 学生在成绩页看到的是一张试卷一张卡片，不再是多次提交平铺。
- 每张卡片显示：考试名称、已考次数、最近得分、最高得分、最近提交时间。
- 点击某张试卷卡片后，进入该试卷的历次作答列表页。
- 在历次作答列表页，能看到每次提交的时间、得分、正确/错误题数。
- 点击某一次提交，进入原来的成绩详情页查看逐题做题情况。

## 3. 前端实现

### 3.1 Android App

- 页面：
  - `ResultsFragment` 重构为按 `examId` 聚合展示。
  - 新增 `ExamHistoryActivity` 作为试卷历次作答页。
  - `ResultDetailActivity` 保持原样，继续查看某一次详细做题情况。
- 组件：
  - 新增 `item_exam_result_card.xml` 试卷聚合卡片。
  - 新增 `activity_exam_history.xml` 试卷历次作答页布局。
  - 新增 `item_exam_history.xml` 历次提交条目卡片。
- 路由：
  - `ResultsFragment` → 点击试卷 → `ExamHistoryActivity`（传入 `exam_id`）
  - `ExamHistoryActivity` → 点击某次提交 → `ResultDetailActivity`（传入 `result_id`）
- 接口请求：
  - `ResultsFragment` 现在同时调用 `getMyResults()` 和 `getStudentExams()`，获取试卷名称。
  - `ExamHistoryActivity` 通过 `getMyResults()` 过滤该试卷的所有提交。
- 已注册新 Activity 到 `AndroidManifest.xml`。

### 3.2 网页端 (Vue)

- 页面：
  - `StudentResults.vue` 重构为按 `examId` 聚合展示。
  - 新增 `StudentExamHistory.vue` 作为试卷历次作答页。
  - `StudentResultDetail.vue` 保持不变。
- 组件：
  - `StudentResults` 新卡片布局显示考试名称、已考次数、最近/最高得分。
  - `StudentExamHistory` 列表展示历次提交记录。
- 路由：
  - 新增 `/student/results/exam/:examId` → `StudentExamHistory`
  - `StudentResults` 点击试卷卡片 → `router.push(/student/results/exam/${examId})`
- 状态管理：
  - 继续复用 `useExamStore`，`loadMyResults()` 和 `loadStudentExams()` 负责数据加载。
- 样式：
  - 新的结果卡片保持与现有 Element Plus 风格一致。

## 4. 后端实现

- Controller：无改动。
- Service：无改动。
- Mapper：无改动。
- Entity / DTO / VO：无改动。
- 配置：无改动。

本次改动完全在客户端完成聚合逻辑，不需要后端配合。

## 5. 数据库与数据流

数据库无新增表或字段。

数据流如下：

1. 客户端调用 `/api/results/my` 获取学生所有成绩记录。
2. 客户端调用 `/api/exams/student` 获取考试信息。
3. 客户端在前端按 `examId` 分组聚合。
4. 显示聚合卡片，点击后根据 `examId` 过滤该试卷的提交记录。

## 6. 接口说明

| 接口 | 方法 | 作用 | 主要参数 | 返回结果 |
|---|---|---|---|---|
| `/api/results/my` | GET | 获取学生所有成绩记录 | 无 | `List<ExamResult>` |
| `/api/exams/student` | GET | 获取学生考试列表 | 无 | `List<Exam>` |
| `/api/results/{resultId}` | GET | 获取某次成绩详情 | `resultId` | `ResultDetail` |

## 9. 风险与后续优化

- 当前聚合逻辑在客户端完成，如果学生成绩记录量极大（超过数千条），建议后续在后端增加聚合接口以减少数据传输量。
- 网页端的导出功能仍导出全部原始提交记录，后续可考虑增加按试卷导出的选项。
- `allowRetake` 的变化不影响成绩页聚合逻辑，成绩页永远按实际提交记录聚合，不受当前是否允许重考影响。
