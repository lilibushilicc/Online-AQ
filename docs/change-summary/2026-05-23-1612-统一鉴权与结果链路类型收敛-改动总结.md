# 统一鉴权与结果链路类型收敛改动总结

## 1. 改动目标

本次改动解决三个维护性和稳定性问题：

1. 后端管理员接口之前依赖控制器里重复调用 `requireAdmin()` 或 `checkAdmin()`，分散且容易漏。
2. 成绩详情、错题本、导出链路大量使用 `Map<String, Object>` 组装返回值，字段边界不清晰，安全审查和后续维护成本高。
3. 前端当前存在 `vue-tsc` 类型错误，导致 `npm run build` 虽然能产出 `dist`，但整体构建状态不干净。

本次目标是把这三处一起收口：统一鉴权入口、把高频结果链路收敛成明确 VO、让前端构建重新全绿。

## 2. 用户视角

- 管理员继续正常使用题库、考试、成绩、配置、文件、反馈等后台功能。
- 学生访问学生端功能不受影响。
- 成绩详情、错题本、导出结果的展示内容不变，但系统内部的数据结构更稳定。
- 前端项目重新可以通过完整构建校验，后续开发时类型问题更容易被提前发现。

## 3. 前端实现

- 页面：
  - `AdminDashboard.vue` 修正并发加载与分数分布图的类型问题。
  - `AdminQuestions.vue` 补齐缺失的编辑弹窗状态与保存逻辑，修复模板引用未定义变量。
  - `StudentExams.vue` 修正 `historyExams` 的类型守卫，避免 `undefined` 推断报错。
- 组件：
  - 无新增组件。
- 路由：
  - 无变动。
- 状态管理：
  - 无结构性变更。
- 接口请求：
  - 继续复用现有 API 函数，前端主要是类型修正，不改协议。
- 样式与布局：
  - 无样式层面的结构调整。

## 4. 后端实现

- Controller：
  - 新增 `@AdminOnly` 注解。
  - `JwtInterceptor` 在解析 JWT 后，直接识别控制器方法或类上的 `@AdminOnly` 注解并校验角色。
  - 删除原先散落在多个控制器中的重复 `requireAdmin()` / `checkAdmin()` 调用。
- Service：
  - `ResultService` 的成绩详情、错题列表、待评分列表改为返回明确 VO。
  - `WrongNotebookService` 的错题本详情、错题本列表改为返回明确 VO。
  - `ExportService` 改为消费强类型 VO，而不再读取 `Map<String, Object>`。
- Mapper：
  - 无变动。
- Entity / DTO / VO：
  - 新增 `ResultDetailVO`、`ResultAnswerVO`
  - 新增 `WrongNotebookDetailVO`、`WrongNotebookSummaryVO`
  - 新增 `WrongQuestionGroupVO`、`WrongQuestionItemVO`
  - 新增 `ReviewItemVO`
- 配置：
  - `JwtInterceptor` 承担统一管理员角色校验职责。

## 5. 数据库与数据流

数据库无新增表字段。

### 5.1 鉴权数据流

1. 请求进入 `/api/**`。
2. `JwtInterceptor` 解析 Token，写入 `userId` 和 `role`。
3. 若目标方法或控制器标记了 `@AdminOnly`，拦截器直接校验 `role=admin`。
4. 非管理员在进入控制器之前就被拒绝。

### 5.2 结果链路数据流

1. `ResultService` 查询 `ExamResult`、`StudentAnswer`、`Question`。
2. 服务层把数据组装成 `ResultDetailVO`、`WrongQuestionGroupVO`、`ReviewItemVO` 等强类型对象。
3. `ExportService` 直接读取这些 VO 字段导出 Excel / Word。
4. 前端按原有字段结构消费，但后端不再依赖脆弱的 `Map<String, Object>` 键名拼装。

## 6. 接口说明

| 接口 | 方法 | 作用 | 主要参数 | 返回结果 |
|---|---|---|---|---|
| `/api/exams` | GET/POST/PUT/DELETE | 考试管理 | 考试参数 | 统一由 `@AdminOnly` 校验管理员权限 |
| `/api/questions/**` | 多种 | 题库管理 | 题目参数 | 统一由 `@AdminOnly` 校验管理员权限 |
| `/api/results/{resultId}` | GET | 成绩详情 | `resultId` | `ResultDetailVO` |
| `/api/results/wrong-questions` | GET | 学生错题列表 | 无 | `List<WrongQuestionGroupVO>` |
| `/api/results/pending-review` | GET | 待评分列表 | 无 | `List<ReviewItemVO>` |
| `/api/wrong-notebooks` | GET | 错题本列表 | 无 | `List<WrongNotebookSummaryVO>` |
| `/api/wrong-notebooks/{notebookId}` | GET | 错题本详情 | `notebookId` | `WrongNotebookDetailVO` |

## 9. 风险与后续优化

- 当前 `@AdminOnly` 已经把管理员鉴权统一到拦截器层，但如果后续权限模型继续复杂化，可以再进一步演进到更细粒度的角色/资源权限注解。
- 结果链路中的高频 `Map<String, Object>` 已经收敛，但项目里仍有其他模块（如公告统计、配置统计、文件上传结果等）继续使用 `Map`，后续可按优先级逐步 VO 化。
- 前端构建已经恢复全绿，后续建议在新增页面时优先补齐显式类型，避免再次出现模板依赖未定义变量的问题。
