# 考试未完成提交与移动端会话对齐改动总结

## 1. 改动目标

原有 Web 学生端要求必须完成全部题目后才能提交，而 Android 学生端已经支持“未作答也可提交”的确认流程，双端交互不一致。同时 Android 端考试提交仍沿用旧的 `studentId` 请求模型，没有跟随后端最新的 `attemptId` 会话机制。此次改动统一双端提交规则：允许未完成提交，但提交前明确提示未作答题数量，并把 Android 提交流程对齐到 `attemptId`。

## 2. 用户视角

学生在网页端作答时，即使还有题目未完成，也可以点击“提交试卷”。系统会弹出确认框，提示还有多少道题未作答，并说明这些题将按未作答处理；学生确认后即可提交。

Android 学生端继续保留原有未作答确认弹框，用户体验与网页端保持一致。

如果全部题目都已作答，双端仍会显示普通提交确认；如果考试时间到达，系统仍会自动交卷。

## 3. 前端实现

- Web 页面：`StudentExamDetail.vue` 去掉“未答完禁止提交”的硬拦截，改为根据未作答数量生成不同的提交确认文案。
- Web 交互：未作答时提示“这些题目将按未作答处理”，全部作答时沿用普通提交确认。
- Android 页面：`ExamDetailActivity` 继续保留未作答确认弹框逻辑。
- Android 数据模型：`ExamDetail` 增加 `attemptId` 字段，`SubmitExamRequest` 改为提交 `attemptId + useTime + answers`。
- Android 接口请求：考试详情读取 `attemptId`，提交答卷时不再传旧的 `studentId`。

## 4. 后端实现

- Controller：无新增接口，继续复用考试详情与提交接口。
- Service：无新增业务分支，仍按现有评分流程处理空答案。
- Mapper：无改动。
- Entity / DTO / VO：无改动。
- 配置：无改动。

## 5. 数据库与数据流

学生进入考试后，后端继续通过 `attemptId` 标识当前作答会话。

无论题目是否全部完成，前端确认提交后都会把当前答案列表提交给后端；未作答题在请求体中以空字符串形式传输，后端按空答案记入 `student_answer`，并统一参与判分、错题统计与成绩生成。

Android 端现在也使用与 Web 相同的 `attemptId` 会话数据流，避免出现旧模型与后端不一致的问题。

## 6. 接口说明

| 接口 | 方法 | 作用 | 主要参数 | 返回结果 |
|---|---|---|---|---|
| `/api/exams/{examId}` | GET | 获取考试详情与当前作答会话 | `examId` | `exam`、`questions`、`attemptId` |
| `/api/exams/{examId}/submit` | POST | 提交未完成或已完成的答卷 | `attemptId`、`useTime`、`answers` | `resultId`、得分统计 |

| 文件 | 作用 |

## 8. 验证方式

```sh
cd Oline-AQ-vue
npm run build

cd ..\Oline-AQ-app
.\gradlew.bat :app:assembleDebug
```

手工验证重点：

1. Web 学生端保留若干题未作答后点击提交，确认会弹出未作答提示，确认后可成功交卷。
2. Android 学生端保留若干题未作答后点击提交，确认会弹出未作答提示，确认后可成功交卷。
3. 双端全部题目作答后点击提交，确认仍可正常交卷。
4. 时间到自动交卷流程不受本次改动影响。

## 9. 风险与后续优化

- 当前 Web 与 Android 都只提示未作答题数量，未额外列出具体未作答题号；如果后续需要更强引导，可在确认框中附加题号列表。
- Android 当前仍未实现与 Web 一样的考试退出确认与草稿恢复统一体验，如需继续对齐，可在后续迭代中补齐。
- 若未来需要按考试配置决定“是否允许未完成提交”，可在考试设置中增加开关并由双端读取后动态控制。
