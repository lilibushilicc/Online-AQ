# 考试重考与稳定乱序映射修复改动总结

## 1. 改动目标

本次改动解决两个核心问题：

1. 前后端虽然暴露了“允许重考”能力，但数据库唯一约束和前端会话防重会阻止第二次提交，导致功能名义存在、实际不可用。
2. 考试开启题目乱序或选项乱序后，草稿恢复依赖下一次重新随机的详情接口，刷新页面或换设备恢复时会出现题目顺序、选项顺序与上次不一致的问题，进而造成答案错位。

因此本次改动把考试作答流程正式升级为“按作答会话 attemptId 管理”，让重考、草稿、评分、成绩回放都基于同一会话闭环运行。

## 2. 用户视角

- 学生在允许重考的考试中，提交一次后仍可再次进入并重新作答。
- 每次作答都会形成独立成绩记录，成绩列表和历史页可正常看到多次提交。
- 如果考试开启了题目乱序或选项乱序，学生刷新页面、关闭后再次进入、甚至换设备继续作答，题目顺序和选项顺序都保持一致，不会出现草稿答案错位。
- 草稿仍然自动保存，提交后会自动切换到下一次新的作答会话。

## 3. 前端实现

- 页面：
  - `StudentExamDetail.vue` 增加 `attemptId` 会话状态。
  - 草稿保存、草稿恢复、正式提交全部附带 `attemptId`。
  - 去掉按 `examId` 的前端硬防重逻辑，改为允许同一考试多次提交。
  - 已答题统计改为按非空答案计算，避免空字符串被算作已完成。
- 组件：
  - 不新增组件，继续复用原答题页结构。
- 路由：
  - 无新增路由。
- 状态管理：
  - `useExamStore.submitExam()` 改为提交 `attemptId`。
- 接口请求：
  - `getExamDetailApi()` 支持传入可选 `attemptId`。
  - `saveDraftApi()`、`submitExamApi()` 请求体新增 `attemptId`。
  - `loadDraftApi()` 返回值补充 `attemptId`、`shuffleSnapshot`。
- 样式与布局：
  - 无样式结构性变更，沿用现有答题页布局。

## 4. 后端实现

- Controller：
  - `ExamController` 的考试详情、草稿保存、草稿读取、草稿清除接口补充会话逻辑。
  - 学生访问考试详情或草稿接口时，先校验是否被分配到该考试。
- Service：
  - `ExamService` 新增 `assertStudentCanAccessExam()`、`getOrCreateDraft()`，并在学生端详情接口中创建或复用当前作答会话。
  - 新增稳定乱序快照 `StableShuffleSnapshot`，保存题目顺序和每道题的选项映射。
  - `ResultService` 提交时按 `attemptId` 和乱序快照进行答案还原与评分，提交成功后自动生成下一次重考会话。
- Mapper：
  - 继续复用 `DraftAnswerMapper`、`ExamResultMapper`、`StudentAnswerMapper`，未新增新 Mapper。
- Entity / DTO / VO：
  - `SubmitExamRequest` 新增 `attemptId`，去掉不安全的 `studentId` 提交语义。
  - `DraftAnswer`、`ExamResult`、`StudentAnswer` 新增 `attemptId`。
  - `DraftAnswer` 新增 `shuffleSnapshot`。
- 配置：
  - `SchemaInitializer` 补充新字段、索引，并移除旧的 `exam_result(exam_id, student_id)` 唯一约束。

## 5. 数据库与数据流

### 5.1 表结构变化

- `draft_answer`
  - 新增 `attempt_id`
  - 新增 `shuffle_snapshot`
- `exam_result`
  - 新增 `attempt_id`
  - 删除旧唯一约束 `uk_exam_result_exam_student`
- `student_answer`
  - 新增 `attempt_id`

### 5.2 数据流

1. 学生进入考试详情：服务端创建或读取 `draft_answer`，生成当前 `attemptId`。
2. 如果考试启用了乱序：首次进入时生成稳定乱序快照并存入 `shuffle_snapshot`。
3. 学生保存草稿：前端携带 `attemptId`、答案、已用时间写回同一条草稿记录。
4. 学生提交：后端读取草稿中的乱序快照，把界面答案映射回原始题目语义后评分。
5. 系统写入一条 `exam_result` 和多条 `student_answer`，都绑定当前 `attemptId`。
6. 提交成功后重置草稿，生成新的 `attemptId`，作为下一次重考会话。

## 6. 接口说明

| 接口 | 方法 | 作用 | 主要参数 | 返回结果 |
|---|---|---|---|---|
| `/api/exams/{examId}` | GET | 获取考试详情 | `attemptId` 可选 | `ExamDetail`，新增 `attemptId` |
| `/api/exams/{examId}/draft` | PUT | 保存当前草稿 | `{ attemptId, answers, useTime }` | `ApiResponse` |
| `/api/exams/{examId}/draft` | GET | 读取当前草稿 | 无 | `DraftAnswer`，含 `attemptId`、`shuffleSnapshot` |
| `/api/exams/{examId}/submit` | POST | 提交当前作答 | `{ attemptId, answers, useTime }` | `{ resultId, totalScore, correctCount, wrongCount }` |
| `/api/results/{resultId}` | GET | 查看某次成绩详情 | `resultId` | 按 `attemptId` 回放本次提交明细 |

## 8. 验证方式

```sh
cd Oline-AQ-spring
.\mvnw.cmd -DskipTests compile

cd ..\Oline-AQ-vue
npm run build
```

验证结果：

- 后端 `mvn compile` 通过。
- 前端 `vite build` 成功产出 `dist`。
- 前端 `vue-tsc --build` 仍存在仓库内原有类型错误，当前观察到的报错位于：
  - `src/views/admin/AdminDashboard.vue`
  - `src/views/admin/AdminQuestions.vue`
  - `src/views/student/StudentExams.vue`

这些错误不由本次 `attemptId` / 重考 / 稳定乱序改动引入，但会影响 `npm run build` 的类型检查阶段返回非零退出码。

## 9. 风险与后续优化

- 当前稳定乱序快照保存在 `draft_answer.shuffle_snapshot`，结构简单、便于恢复，但长期看可以考虑单独抽成更明确的作答会话表。
- 提交成功后，前端仍会主动调用一次清草稿接口；当前逻辑可工作，但后续可进一步统一为完全由后端在提交后接管草稿生命周期。
- 目前本次只修复了重考与稳定乱序映射闭环，管理员角色鉴权、考试详情答案脱敏、SSE 广播范围等高风险问题仍建议继续跟进。
