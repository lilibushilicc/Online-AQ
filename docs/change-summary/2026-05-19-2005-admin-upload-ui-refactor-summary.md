# 上传试题页 UI 与逻辑调整改动总结

## 1. 改动目标

本次改动用于优化管理员“上传试题”页面的界面结构与前端代码组织。原页面把拖拽上传、解析模式、文件记录和格式说明串在一起，信息密度高但层次弱，页面内也存在多处分散的模式判断和状态文案，维护成本偏高。

## 2. 用户视角

管理员进入上传试题页后，会先看到更清晰的导入区与解析策略区：左侧负责选择文件、填写分类、执行上传，右侧直接说明当前解析模式的适用场景与注意事项。

上传后，页面下方继续展示最近导入记录，格式参考区改成可直接浏览的三张示例卡片，不需要逐个展开折叠项即可核对题库格式。

## 3. 前端实现

- 页面：
  - `AdminUpload.vue` 重构为四个区域：导入文件、解析策略、最近导入记录、格式参考。
- 组件：
  - 继续复用 `AdminLayout` 与 Element Plus 组件。
  - 解析模式切换由 `el-switch` 调整为 `el-segmented`，让模式选择更明确。
- 路由：
  - 无变化。
- 状态管理：
  - 仍使用 `exam` store 提供上传解析、加载文件、删除文件能力。
  - 页面内部将 `useAi` 改为从 `parseMode` 推导的计算属性，减少重复状态。
- 接口请求：
  - 无接口变更，继续调用现有上传与解析接口。
- 样式与布局：
  - 新增上传页局部网格布局、状态卡片、文件预览区和格式示例卡片样式。
  - 保留项目现有纸墨风设计体系，不引入新的全局样式依赖。

## 4. 后端实现

- Controller：无变化。
- Service：无变化。
- Mapper：无变化。
- Entity / DTO / VO：无变化。
- 配置：无变化。

## 5. 数据库与数据流

本次改动不涉及数据库结构调整。数据流保持为：前端上传文件 -> 后端保存原文件 -> 解析试题 -> 写入题库表 -> 前端刷新最近导入记录。

本次只优化前端页面结构与交互表达，没有改变题库入库链路。

## 6. 接口说明

| 接口 | 方法 | 作用 | 主要参数 | 返回结果 |
|---|---|---|---|---|
| `/api/files/upload` | POST | 上传原始题库文件 | `file` | `fileId`、文件信息 |
| `/api/files/{fileId}/parse` | POST | 解析已上传文件 | `category`、`useAi` | `questionCount` |
| `/api/files` | GET | 查询已解析文件记录 | 无 | 文件列表 |
| `/api/files/{fileId}` | DELETE | 删除文件及关联题目 | `fileId` | 通用成功响应 |

验证结果：类型检查通过，Vite 生产构建通过，生成新的 `AdminUpload` 资源文件，说明组件模板、脚本与样式均可正常编译。

## 9. 风险与后续优化

- 当前上传页仍是单组件承载全部视图，后续如果继续扩展预览、重试、分步导入，可以再拆分为更小的子组件。
- 最近导入记录仍只展示基础字段，后续可以增加“解析方式”“分类”“上传人”等信息，便于运营排查。
- 若后续加入上传预览和解析校验，建议把页面内文件处理逻辑继续下沉为可复用 composable。
