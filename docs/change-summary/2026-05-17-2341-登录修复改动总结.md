# 登录功能修复改动总结

## 1. 改动目标

修复登录功能无法正常使用的问题。用户点击登录后，即使输入正确的账号密码，也看不到正常跳转或错误提示异常。

## 2. 问题根因分析

经全面排查，发现三个关键问题：

### 问题一：后端认证异常处理不当

后端 `GlobalExceptionHandler` 对 `UnauthorizedException` 使用了 `@ResponseStatus(HttpStatus.UNAUTHORIZED)`（HTTP 401），导致 Axios 进入 error 回调而非 response 回调，前端无法正常读取后端的业务错误消息。

### 问题二：前端错误拦截器覆盖错误信息

前端 `request.ts` 的 Axios error 拦截器中，当 `error.response?.status === 401` 时，无条件执行 `clearAuthState()` + `window.location.href = '/login'`，这导致：
- 登录失败时页面被强制跳转到 `/login`
- `ElMessage.error()` 无法显示后端返回的具体错误消息
- 用户只看到空白刷新，不知道具体错误原因

### 问题三：后端密码校验逻辑可优化

`AuthService` 中通过 `username` + `role` 同时查询用户，若用户名存在但角色不匹配，统一返回模糊错误，可优化为逐步校验，便于后续扩展。

## 3. 用户视角

- 输入正确账号密码 + 正确的身份 → 正常跳转到对应页面（教师到 `/admin/dashboard`，学生到 `/student/exams`）
- 输入错误账号或密码 → 页面左上角弹出 Element Plus 错误提示："账号或密码不正确"
- 身份选错（如用 admin 账号选学生身份）→ 弹出 "账号或密码不正确"
- 登录失败时**不再**强制跳转刷新页面，用户可立即重新输入

## 4. 前端实现

### 4.1 `src/utils/request.ts` — Axios 拦截器修复

- **response 拦截器**（成功响应）：逻辑不变，`code === 200` 时返回 `payload`（`{code, message, data}`），否则 `reject(new Error(message))`
- **error 拦截器**（失败响应）：
  - **优先提取后端业务消息**：`error.response?.data?.message` 存在时，`reject(new Error(serverMessage))`，不执行清除状态和跳转
  - **仅对无业务消息的 401 做跳转**：只有 `error.response?.status === 401` 且后端未返回 message 时，才执行 `clearAuthState()` + `window.location.href = '/login'`

### 4.2 `src/views/login/LoginView.vue` — 无改动

原有的 catch 逻辑已正确处理 `Error` 实例的消息：
```ts
catch (error) {
    const message = error instanceof Error ? error.message : '登录失败，请稍后重试'
    ElMessage.error(message)
}
```

## 5. 后端实现

### 5.1 `AuthService.java` — 登录逻辑优化

- 改为**逐步校验**：先查 username → 判断用户是否存在 → 判断密码 → 判断角色
- 所有校验失败统一返回 `"账号或密码不正确"`（安全考虑，不暴露具体哪个字段错误）

### 5.2 `GlobalExceptionHandler.java` — 认证异常处理修复

- `UnauthorizedException` 不再使用 `@ResponseStatus(HttpStatus.UNAUTHORIZED)`
- 改为返回 HTTP 200 + `ApiResponse(code=400, message="错误详情", data=null)`
- HTTP 200 确保 Axios 进入 response 回调，前端能正常读取 `payload.code` 和 `payload.message`
- `code=400` 表示业务错误，前端 response 拦截器中 `payload.code !== 200` 条件触发，`reject(new Error(message))`

## 6. 数据库与数据流

无需数据库变更。登录数据流如下：

```
前端输入(username, password, role)
  → POST /api/auth/login (HTTP 200)
  → 后端查询 sys_user WHERE username = ?
    → 用户不存在 → UnauthorizedException → ApiResponse(code=400, message="账号或密码不正确")
    → 密码错误 → UnauthorizedException → ApiResponse(code=400, message="账号或密码不正确")
    → 角色不匹配 → UnauthorizedException → ApiResponse(code=400, message="账号或密码不正确")
    → 全部校验通过 → 生成 JWT Token → ApiResponse(code=200, data={token, userId, username, realName, role})
  → 前端 response 拦截器
    → code=200 → 返回 payload → store 解构 data → 保存 token + user 到 localStorage → 跳转
    → code=400 → reject(new Error(message)) → LoginView catch → ElMessage.error(message)
```

## 7. 接口说明

| 接口 | 方法 | 作用 | 主要参数 | 返回结果 |
|---|---|---|---|---|
| `/api/auth/login` | POST | 用户登录 | `{ username, password, role }` | 成功：`{code:200, data:{token,userId,username,realName,role}}`；失败：`{code:400, message:"账号或密码不正确"}` |

## 8. 关键文件

| 文件 | 作用 | 改动类型 |
|---|---|---|
| `Oline-AQ-spring/.../service/AuthService.java` | 登录认证逻辑，改为逐步校验 | 修改 |
| `Oline-AQ-spring/.../config/GlobalExceptionHandler.java` | 全局异常处理，认证异常改为返回 HTTP 200 | 修改 |
| `Oline-AQ-vue/src/utils/request.ts` | Axios 请求/响应拦截器，优先提取后端业务错误消息 | 修改 |

## 9. 验证方式

启动后端和前端：

```bash
# 启动后端 (Oline-AQ-spring 目录)
mvn spring-boot:run

# 启动前端 (Oline-AQ-vue 目录)
npm run dev
```

验证场景：
1. 打开 `http://localhost:5173` → 自动跳转 `/login`
2. 使用默认账号 `admin / 123456` + 身份「教师」→ 点击登录 → 跳转到教师仪表盘
3. 使用学生账号 `2023001 / 123456` + 身份「学生」→ 点击登录 → 跳转到学生考试列表
4. 输入错误密码 `admin / wrong` + 身份「教师」→ 点击登录 → 弹出错误提示 "账号或密码不正确"
5. 使用 admin 账号 + 身份「学生」→ 点击登录 → 弹出错误提示 "账号或密码不正确"
6. 使用不存在的账号 `xxx / 123456` → 弹出错误提示 "账号或密码不正确"

## 10. 风险与后续优化

- **当前实现**：登录失败时所有错误统一提示 "账号或密码不正确"，不暴露具体哪个字段错误，符合安全规范
- **后续可优化**：
  - 密码加密存储（如 BCrypt），`AuthService` 中逐步校验逻辑已为密码加密预留了扩展点
  - 登录失败次数限制与验证码
  - Token 刷新机制
  - 退出登录时调用后端接口使 Token 失效
