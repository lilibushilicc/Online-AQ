# 数据库内容与安全优化方案

## 1. 目标

清理当前 `exam_system` 数据库中的数据质量问题，修复安全隐患，规范数据存储方式。

## 2. 现状分析

当前数据库环境（PostgreSQL `exam_system`，21 张表）存在以下问题：

### 2.1 数据完整性问题

| 问题 | 严重程度 | 说明 |
|---|---|---|
| exam 5/6/7 引用不存在的题目 | **高** | `exam_question` 中 exam_id=5,6,7 引用 question_id 4~13，但 `question` 表中仅有 ID 1,2,3 和 29~48，无 4~13。学生参加这些考试时拿不到题目或报错 |
| 题目内容重复 | 中 | question_id 29~38 与 39~48 内容完全一致（Spring 框架单选题+简答题各 5 道），疑似重复导入 |
| 部分用户数据不规范 | 低 | 用户 ID 4/6/7 的 `real_name` 为 `'1'` 占位符 |

### 2.2 安全风险

| 问题 | 严重程度 | 说明 |
|---|---|---|
| sys_config 明文凭据 | **高** | R2 Cloudflare（endpoint/access_key/secret_key）、DeepSeek API Key、Gmail SMTP 密码均明文存储在 `sys_config` 表和 `init.sql` 中 |
| 凭据提交到 Git | **高** | `init.sql` 和 `release/package/init.sql` 包含真实密钥，存在泄露风险 |

### 2.3 当前数据统计

- **用户**: 11 人（1 admin + 10 学生）
- **考试**: 9 场（exam_id 2,3,5,6,7,8,9,10,11）
- **题目**: 23 道（single + short_answer）
- **考试-题目关联**: 92 条
- **考试成绩**: 13 份
- **学生答案**: 211 条
- **草稿**: 2 份
- **系统配置**: 19 项

## 3. 清理方案

### 第一阶段：修复数据完整性问题

#### A. 清理孤立 exam_question 记录

```sql
-- 删除 exam 5/6/7 中指向不存在题目的关联
DELETE FROM exam_question
WHERE exam_id IN (5, 6, 7)
  AND question_id NOT IN (SELECT question_id FROM question);
```

预期影响：
- exam 5（草稿）：清理 10 条无效关联，剩 3 条（question_id 1,2,3）
- exam 6（已发布但无人作答）：清理 10 条无效关联，剩 3 条
- exam 7（已有成绩）：清理 10 条无效关联，剩 3 条；已存在的 `exam_result` 记录保留，但评分可能不准

#### B. 为 exam 7 补充缺失题目（推荐方案）

由于 exam 7 已有学生作答（result_id 2 和 4）且 assign_all=true，建议从现有题目中选取补充，保持考试可用。

### 第二阶段：清理重复数据

```sql
-- 删除重复题目组 39~48（保留 29~38）
-- 同时更新 exam_question 引用
UPDATE exam_question SET question_id = question_id - 10
WHERE question_id BETWEEN 39 AND 48;

DELETE FROM question WHERE question_id BETWEEN 39 AND 48;
```

### 第三阶段：敏感凭据治理

#### A. 从 init.sql 中移除真实凭据

- `init.sql` 和 `release/package/init.sql` 中的 smtp/R2/AI 配置项仅保留占位符
- 真实凭据改为通过 application.yml 环境变量注入

#### B. sys_config 表改造（可选）

新增字段方案：
```sql
ALTER TABLE sys_config ADD COLUMN encrypted BOOLEAN DEFAULT FALSE;
ALTER TABLE sys_config ADD COLUMN masked_value VARCHAR(255);
```

或改用单独表存储敏感配置，后端提供加密存储接口。

### 第四阶段：补充索引建议

当前已有 44 个索引，检查是否需要补充：

- `exam_result` 目前缺 (exam_id, student_id) 联合索引（已有单独列索引，但联合查询更优）
- `draft_answer` 目前有 `uq_draft_exam_student` 唯一约束，但已改为 attemptId 模型，建议添加 (exam_id, student_id, attempt_id) 索引

## 4. 执行优先级

|---|---|---|
| P0 | 清理 exam 5/6/7 孤立关联 | 5 分钟 |
| P1 | 从 init.sql 移除真实凭据 | 10 分钟 |
| P1 | 将凭据改为环境变量注入 | 30 分钟 |
| P2 | 清理重复题目 | 15 分钟 |
| P3 | 补充索引 | 10 分钟 |

## 5. 风险与后续

### 风险
- 清理 exam 7 的 exam_question 会影响已有成绩的统计结果
- 重复题目可能被前端路由或报表功能引用（需检查）

### 后续可优化方向
- 引入 Flyway 或 Liquibase 管理数据库版本迁移（替代 SchemaInitializer Java 代码方式）
- SMTP/API Key 等凭据上线后使用 Vault 或 AWS Secrets Manager
- 建立数据备份和恢复定期演练流程
