# 三端功能对齐清单

## 总体说明

当前学生侧三端能力可以概括为一句话：

**主流程基本已经打通，但三端的一致性主要体现在“功能结果一致”，还没有完全做到“页面体验一致”或“实现细节一致”。**

更具体地说：

- **网页端桌面** 是当前最完整、最稳定、功能覆盖最全的一套学生端实现，可以视为主参考版本。
- **网页端手机** 现在已经能够复用网页端的大部分能力，学生在手机浏览器里访问时，考试、成绩、练习、错题本、个人中心等主流程都能走通；但它本质上仍然是“移动端外壳 + 桌面业务页复用”，所以在考试详情、成绩详情、错题详情这类页面上，交互体验仍偏桌面化。
- **Android 学生 App** 在考试、成绩、练习、错题本、个人中心、公告、反馈这些主流程上已经覆盖得比较完整，可以承担独立移动端使用场景；但从现有文档和代码线索看，仍有少数网页端细节能力没有完全确认是否同步实现，例如练习历史、导出能力、服务端草稿恢复等。

因此，这份清单的核心目的不是简单判断“有没有功能”，而是把三端现状拆成三类问题：

1. **已经对齐的能力**：三端都能完成同样的业务结果，这部分优先视为可验收。
2. **功能有了但体验没完全对齐**：重点在网页端手机，属于后续移动端优化范围，而不是功能缺失。
3. **还需要继续确认的能力**：重点在 Android App，要先确认有没有实现，再决定是否补做。

从推进顺序上看，建议先做“能力确认”，再做“体验优化”：

- 第一步先确认 Android 端几个未明确项，避免把“文档没写清”误判成“功能没做”。
- 第二步再优化网页端手机的重页面，如考试详情、成绩详情、错题详情。
- 第三步把最终结论沉淀成一份可复用的三端验收记录，供后续版本持续对照。

如果把这三端的关系说得更直接一些：

- **桌面网页端** 是功能基线。
- **手机网页端** 是同功能的移动访问形态。
- **Android App** 是学生侧独立移动产品形态。

所以后续判断一个功能是否“适配完成”时，建议同时问三个问题：

1. 三端是否都能进入这个功能？
2. 三端是否都能得到相同的业务结果？
3. 移动端是否因为直接复用桌面布局而影响使用效率？

只要把这三个问题分开看，就能更清楚地区分“缺功能”“缺确认”和“缺体验优化”。

## 1. 目标

梳理 `网页端桌面`、`网页端手机`、`Android 学生 App` 三端学生功能的对齐情况，明确：

1. 哪些能力已经完全一致。
2. 哪些能力功能一致但交互体验仍有差异。
3. 哪些能力尚未完全确认或需要后续补齐。

本清单用于后续产品验收、移动端优化和 Android 功能补齐时直接对照执行。

## 2. 判断口径

为避免把“功能缺失”和“体验不同”混在一起，本清单统一使用以下判断标准：

- **已对齐**：三端都具备同一项核心能力，且能完成相同业务结果。
- **部分对齐**：核心能力在两端或三端中存在，但有一端仍未明确确认，或结果一致但入口/流程明显不同。
- **功能一致但体验不一致**：能完成同样业务，但布局、操作路径、交互效率明显不同。
- **未确认**：当前仅有文档或零散代码迹象，尚不能直接判定已经落地。

## 3. 当前总体结论

### 3.1 网页端桌面 vs 网页端手机

- 功能层面：基本一致。
- 原因：手机网页端通过 `StudentLayoutWrapper.vue` 切换到 `StudentMobileLayout.vue`，但业务页面本身继续复用原有学生端路由、接口和组件。
- 限制：考试详情、成绩详情、错题详情等页面虽然功能可用，但仍保留大量桌面式 Element Plus 布局，移动端体验未完全专门化。

### 3.2 网页端 vs Android App

- 核心学生主流程基本齐全：考试、成绩、练习、错题本、个人中心、公告、反馈等主链路都有覆盖。
- 限制：部分网页端细节能力在 Android 文档中未完全明确，如练习历史、服务端草稿恢复、导出能力等，需要继续核实或补齐。

## 5. 已完全一致的能力

以下能力当前可以认为三端主流程已对齐：

- 登录
- 考试列表
- 考试详情作答
- 倒计时与自动交卷
- 成绩列表与成绩详情
- 历次作答历史
- 在线练习
- 错题本列表与错题本详情
- 错题加入/移除错题本
- 个人中心
- 系统公告
- 题目纠错反馈

## 6. 功能一致但体验不一致的能力

### 6.1 网页端手机考试详情页

当前手机网页端虽然能正常作答，但 `StudentExamDetail.vue` 仍然保留桌面页面结构特征：

- 右侧 `aside` 答题导航区域仍按桌面思路布局。
- 大量 `el-card`、`el-descriptions`、`el-dialog` 直接复用桌面交互。
- 没有单独的移动端考试详情组件。

结论：

- 功能已对齐。
- 体验仍属于“移动壳 + 桌面业务内容复用”，还不是彻底的手机专属页面。

### 6.2 网页端手机成绩详情 / 错题详情

和考试详情类似，这些页面功能上可用，但依然是桌面组件直接在手机壳中展示，交互密度偏高、信息层级仍偏桌面化。

## 7. 尚未完全确认或建议继续核实的能力

### 7.1 Android 练习历史

- 网页端已有练习历史页面。
- Android 文档当前重点描述了练习功能本身，但未在模块说明中明确写出独立“练习历史”入口与页面。
- 需要后续直接核对 App 页面和 API 调用。

### 7.2 Android 导出能力

- 网页端已支持成绩、错题本、试卷等导出。
- Android 文档中没有把导出列为当前学生端能力重点。
- 需要确认 App 是否已实现，或者明确标记为网页端专属功能。

### 7.3 Android 服务端草稿恢复

- 网页端当前已完成 `attemptId + 服务端草稿 + 稳定乱序快照`。
- Android 文档未明确说明是否接入这一套服务端草稿恢复机制。
- 需要核对 App 的考试详情 API 使用方式以及本地保存策略。

## 8. 建议验收方式

建议按“入口是否存在、接口是否共用、结果是否一致、体验是否明显退化”四个维度验收：

| 维度 | 验收问题 | 说明 |
|---|---|---|
| 入口 | 三端是否都能进入该功能？ | 例如底部导航、列表按钮、详情跳转 |
| 接口 | 是否复用同一类后端接口？ | 用于判断是同能力还是近似能力 |
| 结果 | 最终业务结果是否一致？ | 如是否都能提交考试、查看历史、加入错题本 |
| 体验 | 是否存在明显移动端障碍？ | 如布局拥挤、操作层级深、关键按钮距离热区过远 |

## 9. 后续建议优先级

### 第一优先级：明确 Android 未确认项

1. 核对 Android 是否有练习历史。
2. 核对 Android 是否有导出能力。
3. 核对 Android 是否接入服务端草稿恢复。

### 第二优先级：优化网页端手机体验

1. 为 `StudentExamDetail.vue` 拆分移动端专用布局。
2. 为成绩详情、错题本详情提供更适合手机的分段式展示。
3. 缩减桌面组件在手机壳中的密度，减少横向拥挤和弹窗层级。

### 第三优先级：形成验收清单

后续可按以下方式做一次三端验收：

1. 以学生账号从登录开始，分别走考试、成绩、练习、错题、我的五条主路径。
2. 记录每条路径在三端是否具备相同入口、相同核心结果、相近的数据字段。
3. 对“功能一致但体验不同”的项单独打标签，避免误判为功能缺失。

## 10. 优先处理项清单

如果后续要直接进入开发，建议按下面顺序处理：

1. **先核实 Android 未确认项**
说明：这一步决定“是否真的缺功能”，优先级高于网页端体验优化。

2. **再优化手机网页端考试详情页**
说明：这是当前最明显的“功能有了，但体验不够手机化”的页面。

3. **最后统一补做三端验收记录**
说明：把结论沉淀下来，后续每次改移动端时能复用。
